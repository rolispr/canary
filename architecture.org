* Overview
:PROPERTIES:
:CUSTOM_ID: overview
:END:

canary is a TUI library for Guile. Apps are GOOPS classes with a
~view~ method returning a node tree and an ~update~ method returning
~(cons next-self cmd)~. The engine reads input, decodes typed msgs,
walks the tree, dispatches ~update~ through it, renders the result,
and emits the minimal ANSI byte diff to the host terminal.

The model is a Scheme-flavoured Elm/Bubble Tea: msgs in, state
out, cmds for effects. Two generics carry the whole API:

#+begin_src scheme
view   : (self)     -> node
update : (self msg) -> (cons next-self cmd-or-#f)
#+end_src

Live coding falls out of how methods, GOOPS classes, and the cell
diff interact. Re-evaluate a ~define-method~ from a connected REPL
and the next frame uses the new body. Re-evaluate a ~define-class~
and Guile's redefinition protocol migrates the running widgets.

The architecture has two doors into one shared cell model
(~<term>~):

- write path : view tree -> draw cmds -> cells
- parse path : bytes -> ANSI state machine -> cells

And one door out: ~term-diff->ansi~ emits the byte sequence that
takes ~prev-term~ to ~cur-term~. Every frame swaps the slots and
diffs forward.

** Layer responsibilities

*Event loop:*
- ~<engine>~ holds msg queue, backend, theme, global keymap, root
  widget, focus chain, subscriptions, mouse state, log buffer.
- A bell pipe (one byte over a pipe) wakes the loop fiber when a
  msg arrives from any source: input fiber, ~send~ from a cmd
  thunk, ~every~ ticker, SIGWINCH.
- ~send~ is the only thread-safe entry point: queue mutex plus
  bell write.

*Input decoder:*
- Reads raw bytes from the tty, decodes CSI / OSC / DCS, kitty
  keyboard CSI u, bracketed paste, focus-in / focus-out, mouse SGR
  and 1006 modes.
- Emits typed msg records: ~<key>~, ~<mouse>~, ~<paste>~,
  ~<resize>~, ~<focus>~, ~<blur>~.

*Dispatch:*
- Three-level keymap pipeline (scoped innermost, scoped outer,
  global), then raw-key fallthrough to the focused widget.
- Cascade walker descends the rendered tree and calls ~update~ on
  every widget it sees. Leaf-to-root order for key / mouse;
  broadcast for everything else.

*View tree:*
- Layout records are pure-data nodes. Widgets are ~<focusable>~
  GOOPS instances embedded by reference, not by value.
- ~view~ is called twice per msg: once during the cascade to find
  widgets to dispatch on, once after to produce draw cmds.

*Render pipeline:*
- ~view->cmds~ walks the tree by type check and yields ~<draw>~
  records (move, set-face, put-glyph, set-link, set-zone, ...).
- ~render-cmds-to-term!~ replays those cmds into ~cur-term~,
  mutating cells one at a time.
- ~term-diff->ansi~ walks both ~<term>~ snapshots and emits cursor
  moves, SGR changes, OSC 8 / 133 transitions, then changed glyphs.
- The backend swaps the slots so the next frame diffs against this
  one.

*Theme:*
- A ~<theme>~ holds palettes by name; the active palette resolves
  symbolic colour names referenced from ~<face-attrs>~ at render
  time.
- ~cycle-palette~ / ~set-palette~ swap the active name; every
  ~#:fg 'accent~ in the tree recolours on the next frame.

*Cmd effects:*
- Cmds are records, not quoted lists. ~every~ / ~after~ spawn
  fibers tagged with an id; the engine cancels subs whose owning
  widget left the tree, or whose id matches a ~cancel~.

** Event lifecycle

1. *Input reception.* Input fiber reads bytes, runs them through
   the decoder, emits a typed msg.
2. *Queue + wakeup.* Msg appended to ~engine-msg-queue~ under
   mutex; bell pipe rings to wake the loop fiber.
3. *Filter.* If an ~engine-filter~ is set (handy for tests), it
   sees the msg first and can suppress it.
4. *Keymap dispatch.* For ~<key>~ and ~<mouse>~, walk scoped
   keymaps along the focus path (innermost first), then the global
   keymap. First match becomes an action symbol; the raw key is
   consumed.
5. *Cascade.* For keys / mouse: leaf-to-root through the focus
   chain. For everything else: broadcast through every widget in
   the rendered tree. Each widget's ~update~ method dispatches on
   the msg class; non-matching widgets fall through to the global
   catch-all in ~(canary view)~ and return ~(cons self #f)~.
6. *Cmd collection.* The cascade collects every returned cmd into
   a batch.
7. *Cmd execution.* The cmd handler interprets each: spawn or
   cancel subs, mutate engine knobs (~set-title~,
   ~alt-screen~, ~mouse-mode~, ~cursor~, ~set-palette~), route
   ~focus~ to a widget, run user thunks in a fresh fiber, etc.
8. *View + render.* The loop calls ~(view root)~, walks the
   resulting tree to draw cmds, replays into ~cur-term~, diffs
   against ~prev-term~, writes the minimal byte sequence,
   bracketed by ~\e[?2026h~ ... ~\e[?2026l~ when the host honours
   synchronized output.
9. *Slot swap.* ~cur-term~ becomes ~prev-term~ for the next frame.

A msg with no resulting state change and no cmd skips the render
entirely: nothing changed, nothing to diff.

* View tree and cascade
:PROPERTIES:
:CUSTOM_ID: view-tree-and-cascade
:END:

Layout records are plain data. The renderer recognises them by
type check; when it hits a widget, it calls ~(view widget)~ and
recurses into the returned tree.

#+begin_src scheme
(vbox
 (boxed (dired #:path "/foo") #:title "left")
 (hbox  (align (spinner) #:h 'center #:width 20)
        (pad   (button #:label "ok" #:action 'save) #:left 2)
        (width (chat-input app) 40)))
#+end_src

The cascade does the same walk for dispatch. Two passes per msg:

- *Discovery pass.* Walk the rendered tree, collect every widget
  by ~eq?~ identity, build the focus chain.
- *Dispatch pass.* For keys / mouse, call ~update~ on the focus
  chain leaf-to-root. For other msgs, call ~update~ on every
  widget. Each call yields ~(cons next-self cmd)~; the engine
  threads ~next-self~ back into its parent slot.

** Why widgets embed by reference

Passing ~(chat-input c)~ into a container, not ~(view (chat-input
c))~, keeps the widget identity in the rendered tree. The
cascade can only reach widgets it can see by reference; expanding
the widget inline strips its identity and the cascade walks past
it without ever dispatching ~update~. Focus, ~<mount>~, and
~every~ sub-cancellation all key off that identity.

** Catch-all

~(canary view)~ defines a global ~(update self msg)~ catch-all
that returns ~(cons self #f)~. Users specialise ~update~ on the
msg classes they care about; everything else silently no-ops. No
per-class boilerplate.

** Draw cmds

~<draw>~ records carry one cell-grid operation each:

| record           | meaning                                          |
|------------------+--------------------------------------------------|
| ~move~           | move cursor to col, row                          |
| ~set-face~       | switch SGR state (fg, bg, bold, italic, etc.)    |
| ~put-glyph~      | write one codepoint at the cursor               |
| ~set-link~       | start / end OSC 8 hyperlink                     |
| ~set-zone~       | start / end OSC 133 shell-integration zone      |
| ~clear~          | full-screen clear                                |
| ~image~          | inline kitty-graphics image                      |
| ~clickable~      | record region for mouse hit-testing             |

Clickable cmds never reach the tty; the backend consumes them to
populate ~engine-click-regions~.

** Cell model

A cell carries:

- codepoint (one Unicode scalar)
- ~<face-attrs>~: fg, bg, bold, italic, underline style and
  colour, overline, blink, inverse, conceal, strike
- hyperlink uri (OSC 8 metadata)
- semantic-content tag (OSC 133 zone)

Adding a new attribute means extending ~<face-attrs>~ once; both
the write path and the parse path pick it up because both write
into the same cell record.

** Optimisations

- *Static caching.* ~(static node)~ caches rendered draw cmds
  keyed on the rect; expensive sprite or panel chrome renders
  once until its rect changes.
- *Frame diffing.* ~term-diff->ansi~ emits only the cells that
  changed. The common idle frame is zero bytes.
- *Synchronized output.* Mode 2026 brackets the diff so the host
  never paints a half-written frame.
- *Cell-aware clipping.* The diff emitter understands wide
  characters and combining marks; partial wide-cell writes are
  resolved before bytes go out.

** Op records as dispatch targets

The parser emits op records rather than calling mutation
procedures directly. Dispatch is a method on the op class. That
lets callers intercept emulator decisions from the REPL by
specialising on the op:

#+begin_src scheme
(define-method (update self (op <op-set-mode>))
  ;; observe or override mode-set decisions live
  ...)
#+end_src

Same shape as msg dispatch on the app side. The emulator is just
another consumer of multi-method polymorphism.

** Entry points

- ~make-term~, ~term-process-output!~, ~term-process-bytes!~
- ~view->grid~ : render a view tree into a fresh ~<term>~ for
  comparison
- ~term->text-snapshot~, ~term->ansi-snapshot~ : golden files
- ~replay-ansi~ : parse a recorded byte stream
- ~mode-get~ / ~mode-set!~ : poke individual VT modes

** Resolution rules

- Each palette is a name -> hex colour map.
- Faces store either a name symbol or a literal hex string. Names
  resolve through the active palette; literals pass through
  unchanged.
- A name missing from the active palette falls back to the
  default (first declared) palette.
- ~cycle-palette~ rotates by declaration order. ~set-palette~
  takes a specific name. The engine tracks every registered
  palette so users do not maintain a list.

** Dynamic theme swap

Theme changes are a single cmd:

1. ~(cycle-palette)~ or ~(set-palette 'light)~ in any ~update~.
2. Cmd handler mutates ~engine-theme~.
3. Next frame's ~view~ produces the same tree; the renderer
   resolves names against the new palette.
4. ~term-diff->ansi~ emits the changed SGRs as part of the
   ordinary diff.

There is no per-widget theme propagation step: widgets only ever
referenced names.

** Why no per-app default background

Most TUI surfaces want to draw their own background. canary
leaves cells with no explicit ~#:bg~ at the terminal default
(SGR 49) so a host terminal's background shows through. Apps that
want a solid background use ~fill~ at the root of their view.

* Focus and keymap dispatch

** Three levels

| level   | where it lives                                  | use for                                  |
|---------+-------------------------------------------------+------------------------------------------|
| global  | ~engine-keymap~ slot, set via ~run-app #:keymap~ | app-wide binds: quit, ctrl-c            |
| scoped  | ~with-keymap~ wrapper in the view tree           | scene binds, modal overlays, per-pane    |
| raw     | focused widget's ~update (msg <key>)~ method     | typing, inline nav (textinput, menu)     |

Dispatch order on a key:

1. Engine collects scoped keymaps from ~with-keymap~ wrappers
   along the focus path AND inside the focused widget's own view
   (innermost first).
2. Stack walks top-down: scoped innermost, scoped outermost,
   then engine global. First match fires its action symbol as a
   cascade msg; the raw key is consumed.
3. If nothing matches, the raw key routes down the focus chain to
   the focused widget's ~update (msg <key>)~ method.

** Modal overlay shape

A pause menu opens by setting a slot. The scene's ~view~ swaps
the active scoped keymap based on whether that slot is set:

#+begin_src scheme
(define-method (view (c <canvas>))
  (with-keymap (if (canvas-pause-menu c) %pause-km %canvas-km)
    (vbox (map-view c)
          (and (canvas-pause-menu c) (canvas-pause-menu c)))))
#+end_src

The wrapper vanishing from the tree de-scopes the keymap. No
push / pop bookkeeping; the structure of the tree is the
structure of the keymap stack.

** Chord state

Multi-key chords (~(bind #\g #\g 'top)~) hold partial state on
the engine, not on the scoped wrapper. Scoped wrappers are
rebuilt every frame, so chord state would not survive a render.
Chord-bearing binds belong on the global keymap.

* Subscriptions and lifecycle
:PROPERTIES:
:CUSTOM_ID: subscriptions-and-lifecycle
:END:

A widget gets ~<mount>~ when it appears in the view tree and
~<unmount>~ when it leaves. ~<init>~ fires once per app at
startup. The engine tracks mount state by widget identity
(~eq?~), so re-using the same instance across frames keeps it
mounted.

** every / after

#+begin_src scheme
(every #:hz 10 #:id (list 'spin self) (lambda () (tick)))
(after #:ms 500 #:id 'flash         (lambda () 'unflash))
#+end_src

Both spawn a fiber. The thunk's return value becomes a msg on
the queue. ~#:id~ keys the sub for cancellation; re-issuing the
same id is idempotent (the engine spawns once).

When the issuing widget leaves the view tree, the engine cancels
every sub tagged with that widget. Cancellation is automatic for
the common case; users almost never call ~cancel~ explicitly.

** Cmd taxonomy

| cmd                                  | effect                                       |
|--------------------------------------+----------------------------------------------|
| ~#f~                                 | no-op                                         |
| ~'quit~                              | exit ~run-app~                                |
| ~(batch c1 c2 ...)~                  | run in parallel                               |
| ~(sequence c1 c2 ...)~               | run sequentially, await each                  |
| ~(every #:hz N [#:id k] producer)~   | persistent ticker                             |
| ~(every #:ms N [#:id k] producer)~   | same                                          |
| ~(after #:ms N [#:id k] producer)~   | one-shot timer                                |
| ~(cancel id)~                        | stop a sub by id                              |
| ~(println "string" ...)~             | line to scrollback above alt-screen           |
| ~(set-title "name")~                 | runtime OS title change                       |
| ~(clear-screen)~                     | force full repaint                            |
| ~(cursor 'hidden /-visible/-bar/...)~| runtime cursor change                         |
| ~(alt-screen 'on/-off)~              | toggle alt screen                             |

** Hot reload rules

- *Re-evaluating ~define-method~:* method redefinition replaces
  the body in place. The next render or next dispatch picks up
  the new body.
- *Re-evaluating ~define-class~:* Guile runs the class
  redefinition protocol. Existing instances migrate to the new
  slot layout; default values fill new slots.
- *Re-evaluating a theme form:* swap the engine's theme. Next
  frame recolours.
- *Re-evaluating a layout helper:* next call sees the new body.
  Helpers used in cached ~static~ subtrees need a
  ~(clear-screen)~ to repopulate the cache.

** REPL access

~make repl~ runs ~guile -L . --listen=37147~, exposing a TCP
REPL on port 37147. Any client speaking Guile's REPL protocol
can connect: Geiser, ~telnet localhost 37147~, custom tooling.

There is no per-session isolation as in TUIOS / wish: the REPL
is the same image the engine runs in, so REPL evaluations
mutate the live engine. That is the design goal, not a
limitation: edit your app while it runs.

** Tests against the same model

Because ~<term>~ is both the renderer's write target and the
emulator's parse target, tests have one model to assert
against:

#+begin_src scheme
(let ((t (view->grid (vbox (txt "hello") (txt "world"))
                     #:width 20 #:height 5)))
  (term->text-snapshot t))
;; => "hello               \nworld               \n..."
#+end_src

Or feed a byte stream through the parser and assert on the same
shape:

#+begin_src scheme
(let ((t (make-term 80 24)))
  (term-process-bytes! t #vu8(...))
  (term->ansi-snapshot t))
#+end_src

** Naming conventions

Uniform across the library:

- ~<thing>~ is the class or record type
- ~thing~ is the constructor (no ~make-~ prefix)
- ~thing?~ is the predicate
- ~thing-field~ is the accessor

Applies to engine plumbing (~(engine #:backend ...)~,
~(ansi-backend #:port ...)~) and spring helpers
(~(spring-animation ...)~, ~(spring-bouncy)~) too.

* Terminal capabilities

~backend-init~ enables the following modes; ~backend-shutdown~
restores them in reverse order.

| escape         | mode                                                                |
|----------------+---------------------------------------------------------------------|
| ~\e[?1049h~    | alternate screen buffer                                              |
| ~\e[?25l~      | hide cursor                                                          |
| ~\e[?1004h~    | focus reporting (emits ~<focus>~ / ~<blur>~)                         |
| ~\e[?2004h~    | bracketed paste (emits ~<paste>~ with ~paste-text~)                   |
| ~\e[>5u~       | kitty keyboard protocol, flags 1 + 4 (disambiguate + alternate keys) |

Each frame's cell diff is bracketed by ~\e[?2026h~ ... ~\e[?2026l~
(synchronized output) so terminals that honour it never paint a
half-written frame.

Mouse reporting is opt-in via the ~#:mouse~ kwarg on ~run-app~
(~'click~ / ~'cell~ / ~'all~). Kitty graphics support is
capability-probed at init; the renderer falls back to text views
when the host does not speak the protocol.
